# AGENTS.md

この文書は、前段の `汎用ルール` と後段の `本リポジトリ固有ルール` に分けて保守する。

前段の `汎用ルール` は、他リポジトリへ先頭からそのまま転用しやすい共通指針として扱う。

---

## 汎用ルール

### 長期作業タスク継続

- 長期タスクの初回では、`目標`、`完了条件`、優先度付きの小タスク、再開コマンド、基本スモークテスト手順を先に整備し、その後は 1 session で 1 slice ずつ進める。
- セッション開始時は、作業ディレクトリ、直近の git 履歴、進捗ログ、未完了 checklist、起動スクリプト、最低限の動作確認の順で足場を確認し、壊れた基盤を見つけたら新規実装より先に修復する。
- 進捗管理は Markdown の自由文だけに依存せず、機械更新しやすい JSON、key-value、checklist 形式を併用し、項目本文の書き換えより達成状態の更新を優先する。
- セッション終了時は、完了したこと、未完了事項、次の最短アクション、実施した検証、残るリスクを構造化して残し、次の担当が推測で補完しなくてよい状態で止める。
- 完了宣言はコードを眺めただけで行わず、明示した完了条件、関連テスト、必要なら E2E や実運用に近い確認を満たしてから行う。再実行され得る副作用には checkpoint 境界と idempotency を設ける。

### ブランチ・PR・CI完了フロー

- 実装を伴うタスクでは、最初に現在ブランチと作業ツリーを確認する。`main`、`master`、または既定ブランチ上にいる場合は、作業開始前に目的が分かる作業ブランチを作成してチェックアウトする。
- 既に作業ブランチ上にいる場合は、そのブランチを継続してよい。ただし、作業ツリーに未確認の差分がある場合は、差分の所有範囲を把握し、無関係な変更を巻き込まない。
- 作業は「動くところまで」ではなく、要求された成果、関連ドキュメント、テスト、既知の不備修正までを完了条件として扱う。途中で真の blocker が判明した場合は、原因、再現条件、未完了範囲、次の最短アクションを明文化する。
- 変更は意味のある slice ごとに分け、各 slice で関連する確認を行ってからコミットする。コミットメッセージは本書の規定どおり日本語で書き、巨大な未整理差分を最後まで残さない。
- PR を作る前に、ローカルで実行可能な最小十分な検証を済ませる。検証できない項目がある場合は、理由と残存リスクを PR 本文に明記する。
- 作業完了時は作業ブランチを push し、PR を作成または更新する。PR 本文には変更内容、保持した既存挙動、検証結果、未実行項目、リスクを記載する。
- PR 作成だけでは完了扱いにしない。CI がある場合は、最新 head の CI 状態を確認し、失敗していればログを読んで原因を特定し、修正、コミット、push、再確認を繰り返す。
- CI を通せない真の blocker がある場合のみ、完了ではなく blocker として報告する。報告には失敗している check 名、ログ上の根拠、試した修正、残る判断事項を含める。
- 最終報告では、ブランチ名、PR URL、最新 commit、ローカル検証、CI 結果、残るリスクを簡潔に示す。CI が未完了または失敗中の場合は、完了と表現しない。

### コミットメッセージ

- 生成するコミットメッセージは必ず日本語で書く。
- 文体は簡潔に保ち、要点がすぐ伝わる表現を優先する。
- コード記号、ファイル名、クラス名、関数名、メソッド名、一般的な技術用語は必要に応じて英語のままでよい。
- 基本形式は 1 行目に変更内容の要約を書き、補足が必要な場合のみ 2 行目以降に追記する。

### ドキュメント更新ルール

- 実装、挙動、セットアップ、アーキテクチャ、設計を変更したら、`README.md` の更新要否を必ず確認する。
- `README.md` は仕様変更に追従して更新しつつ、README の一般的な役割から逸脱する過剰な追記は避ける。
- 変更や新規設計を行ったら、`docs/` 配下の関連文書を必ず見直し、現状の挙動、制約、意図とずれていれば更新する。

### コーディング原則

### DRY

- 同一ロジック、定数、バリデーションが 2 回現れたら抽出を検討し、3 回現れたら関数、クラス、モジュールのいずれかへ必ず抽出する。
- 設定値は環境変数または設定モジュールに集約し、マジックナンバーを持ち込まない。
- 重複するテストデータは factory や fixture に集約する。
- コピーアンドペーストでの派生修正を避け、共通化後に差分だけを表現する。
- 同じ入力検証を UI と API で別々に増殖させず、責務に応じて共有可能なルールを整理する。
- リファクタリングは、重複箇所の探索、共通化単位の決定、抽出、逆依存の解消、テスト確認の順で進める。

### KISS

- 実装はまず最小構成で成立させ、不要な抽象化や仕組み化を後回しにする。
- 分岐のネストは 3 階層以内を目安にし、巨大関数はおおむね 40〜60 行を超える前に分割を検討する。
- 早期 `return` やガード節を使って認知負荷を下げる。
- DSL、メタプログラミング、反射、動的 import は最後の手段として扱う。
- 依存ライブラリは最小限に保ち、標準機能で足りる場合は追加しない。
- 初見の開発者が 5 分程度で処理の骨子を追える複雑さを目安にする。

### SoC

- 層分割を守り、`UI/Presentation`、`Application/Service`、`Domain`、`Infra/DB` の関心事を混在させない。
- ログ、計測、リトライ、認可などの横断的関心はミドルウェア、デコレータ、共通基盤で注入する。
- フロントエンドでは状態管理と描画を分離する。
- コンポーネント内で `fetch`、DOM 操作、業務ロジックを同居させない。
- ドメイン層から HTTP クライアントや ORM を直接呼ばない。

### SRP

- ファイル名と公開 API が責務を説明できる状態を保ち、曖昧なら分割する。
- クラスは 1 つのユースケースまたは 1 つのポリシーに集中させる。
- モデル定義、入出力、整形、保存処理を 1 ファイルに混在させない。
- God Object、God Component、肥大化した utility モジュールを避ける。

### YAGNI

- Issue や要件に存在しない抽象化、設定、拡張ポイントは追加しない。
- 未使用フラグや将来用の空実装を置かない。
- ただし、セキュリティ、可観測性、エラーハンドリングに必要な最低限の備えは先行してよい。

### POLA

- 同種の関数や API はシグネチャと戻り値の規約を揃え、直感に反する特殊挙動を避ける。
- 非同期 API は常に `async`/`await` または Promise の規約に従い、呼び出し側を驚かせる同期値返却を混在させない。
- 破壊的操作は `delete*` や `mutate*` のように名前で明示し、暗黙の副作用を避ける。
- バリデーションメッセージは入力タイミングと表現を一貫させる。

### OCP

- 種別追加のたびに条件分岐を増殖させるのではなく、ポリモーフィズム、戦略パターン、差し替え可能な構造で拡張できるようにする。
- 支払、翻訳、ストレージなどの外部統合はインターフェースで抽象化し、アダプタ実装を分離する。
- `if type == 'A'` の列挙が伸び続ける構造を放置しない。

### エラーハンドリングと可観測性

- エラーを消すこと自体を目的にせず、なぜその機能が正常に動かなかったのかという本質原因を優先して特定する。
- 失敗は例外または `Result` 型などで明示し、黙殺しない。
- 不具合や不備は適切にエラーとして検知できる状態を重視し、無暗に握りつぶしたり、過剰なフォールバックで問題を隠したりしない。
- 想定可能な失敗は、再試行、フォールバック、ユーザー通知の方針を事前に決める。
- ログは構造化し、`level`、`event`、`context` を含め、PII は必ずマスキングする。
- 主要ユースケースでは成功率、p95 latency、再試行率などのメトリクスを計測できる状態を目指す。

### テスト

- テストピラミッドを意識し、Unit Test を厚く、Integration Test を必要十分に、E2E はクリティカル経路に絞る。
- テスト記述は `Given`、`When`、`Then` の読みやすさを意識する。
- 重複するテストデータは factory や fixture に集約する。
- UI テストは `getByRole` などロールやラベル中心で選択し、`data-testid` は最後の手段とする。
- ネットワーク越しの依存は原則モックし、境界の妥当性は契約テストで補う。
- カバレッジの目安は line 80%、branch 70% とするが、数字だけを目的化しない。
- バグ修正、障害対応、仕様解釈の修正を行った場合は、同種の不具合を再発させない回帰テストの追加を原則とする。
- 回帰テストは、修正後の正常系だけでなく、修正前に何が壊れていたかが分かる失敗条件と期待結果を明示する。
- 回帰テストは実装詳細に過度に依存させず、外から観測できる契約、エラー、ログ、状態遷移を優先して固定する。
- 複数層にまたがる不具合は、最小コストで再発を防げる層から順に Unit Test、Integration Test、E2E のどこへ置くかを判断する。
- 一度起きた設定ミス、境界値バグ、非同期タイミング起因の不具合は、単発修正で終わらせず、将来の変更でも落ちるテストとして残す。

### 依存と構成管理

- 依存は最小限に保ち、未使用依存は削除する。
- ロックファイルは依存変更に追従して更新し、既知の CVE を監視する。
- 設定は 12-factor app の考え方に沿って扱い、秘密情報は環境変数や秘密管理基盤に置く。
- セキュリティリスクを増やさないよう、生成物、秘匿情報、環境依存ファイル、認証情報を誤って追跡しないか常に確認し、必要なら `gitignore` を更新する。
- 公開 API に破壊的変更を入れる場合は、セマンティックバージョニングを意識して扱う。

### スタイルと可読性

- 変数名、関数名、メソッド名、クラス名は短さより可読性と意味の明確さを優先する。
- 略語は一般的に通じるものだけに留める。
- 関数名は副作用の有無が分かるように付ける。
- 新規アサインされたメンバーでも理解しやすいよう、クラス、関数、メソッド、または複雑で個性の強い処理には、意図と責務が分かる明確なコメントを付ける。
- コメントは「なぜ」を補い、「何をしているか」はコードと命名で表現する。
- ソースコード内のコメントに改修時の連絡事項、申し送り、作業メモを残さず、現状の仕様や挙動の説明だけを書く。
- 現在の仕様理解やキャッチアップに役立つ補足は積極的にコメントし、過去の経緯ではなく今どう動くかが分かる内容を優先する。

### 記述原則

- ドキュメントは `DRY`、`KISS`、`SRP`、`SoC`、`OCP`、`POLA` と矛盾しない形で保守する。
- 単一ファイルが肥大化しないよう、役割単位で内容を分割する。
- 1 つの節が概ね 3〜5 項目を超えて膨らむ場合は、別節または別文書への切り出しを検討する。
- 重複説明は共通セクションに集約し、各所では差分だけを書く。
- 魔法値、隠れた前提、暗黙の依存関係を文書に残さない。

### アーキテクチャ記述

- コード構造を説明する場合は、`責務`、`依存`、`分岐条件`、`拡張ポイント` を明示する。
- 説明は簡潔に保ちつつ、別の開発者が責務配置と安全な拡張方法を判断できる粒度にする。

### 分割指針

- コードファイルと長文ドキュメントは、おおむね 600〜800 行を分割検討の目安とする。ただし絶対上限ではない。
- コードを分割する場合は、`service`、`domain`、`IO` など再利用しやすい断面を明示する。

### 変更時チェックリスト

1. `README.md` の更新要否を確認する。
2. 影響を受ける `docs/` 配下の文書を確認する。
3. 必要なら `gitignore` の更新要否を確認する。
4. ルールや作業指針の不備が明らかになった場合は、対応する `AGENTS.md` の更新要否を確認する。
5. 実装や設計の意味が変わった場合は、関連ドキュメントを同じ変更内で更新する。

## 完了の定義
作業は、次のすべてを満たしたときにのみ完了である:
- 要求された成果が実装されている、または真の阻害要因が文書化されている。
- 関連する検証が実行済みである、または未実行理由が明示されている。
- 厳格な自己レビューが完了している。
- 既知の重大問題が未報告のまま残っていない。
- 変更パッチが、慎重なメンテナであれば現実的にマージ可能な品質である。

## 本リポジトリ固有ルール

- 2000行を超えない小規模開発では、原則としてPRを分割せず、可能な限り大きな単位で提出する。
- ルート `AGENTS.md` は進行原則・品質基準・完了条件を定義し、サブディレクトリの `AGENTS.md` は領域固有の実装規約・検証コマンド・注意点を補完する。
- サブディレクトリ文書が存在する場合でも、長大タスクの進行原則（計画作成、blocker 基準、PR 作成条件、停止時整合）は本書に合わせる。
- 計画テンプレートは `plans/TEMPLATE.md` を基準とし、必要ならタスクに応じて項目を拡張する。
- 実装、挙動、セットアップ、設計が変わる場合は `README.md` と `docs/` の更新要否を必ず確認する。
- テスト実装方針は次を標準とする。
  - ロジック変更: まず Unit Test を追加し、境界入力・異常系・回帰条件を優先して固定する。
  - モジュール間連携変更（XR入力、儀式状態遷移、export など）: 必要最小限の Integration Test を追加し、公開契約の整合を確認する。
  - UI/操作フロー変更: クリティカル導線のみ E2E もしくは同等のスモークテストを追加する。
- テストを追加できない場合は、理由・代替検証・残存リスクを PR と最終報告に明記する。
- 不具合修正時は、修正前に失敗する条件を再現する回帰テストを原則同一変更で追加する。
