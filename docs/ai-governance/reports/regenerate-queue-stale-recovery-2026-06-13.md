# UI/UXレビュー報告: 再生成キューの古い進行中カード回復

## 1. 概要

- 対象PR / 作業: Cloud Run revision / instance 切替後も再生成ジョブ状態を復元し、既存の古い進行中カードを補正する
- 変更した画面・コンポーネント: `GenerationQueuePanel`, `useWordPack` の再生成通知
- 判定: Pass
- P0件数: 0
- P1件数: 0
- P2件数: 0

## 2. ユーザー価値

- 対象ユーザー: 本番環境でWordPackを生成・再生成する利用者
- 利用文脈: 再生成完了後に右側の生成キューを見る、または後から同じブラウザで戻る
- ユーザー目的: 完了した生成が「生成中」のまま残っていないか判断し、保存済みWordPackを開く
- 支援するタスク: 長時間残った進行中通知の状態理解、保存済みWordPackの確認、必要時の再生成判断
- このUIが助ける理解・判断・行動: ジョブ状態が完了済みで結果スナップショットを確認できた場合だけ成功カードへ補正し、保存済み結果のプレビューを開ける。確認できない場合は失敗状態と回復手段を示す
- このUIがなければ困る点: 完了した生成が何日も「生成中」に見え、再生成すべきか、結果が消えたのか判断できない
- 削るべき情報・操作: 追加の常時説明や確認ダイアログは入れない。既存のカード構造と操作数を維持する
- 検証仮説・成功指標: 20分を超えた進行中カードが persisted job status の成功確認後に成功へ変わり、ユーザーがプレビューを開ける

## 3. 初見理解

- 何の画面か分かるか: 「生成キュー」見出しと「進行中」「完了」の区分で分かる
- 今どこか分かるか: 右側の生成履歴領域として既存構造を維持
- 何ができるか分かるか: 進行中は「キューから隠す」、完了はカード全体からプレビューを開ける
- 最初の有意味な行動: 古いカードが補正された後、完了カードを開く。または確認不能時の案内に従って一覧更新・再生成を行う
- 操作結果を予測できるか: 成功カードの accessible name は「alpha の生成結果プレビューを開く」で結果が予測できる
- 失敗時に戻れるか: 「ジョブIDが保存されていないため、完了状態を確認できません。保存済みWordPackを一覧で確認するか、必要ならもう一度生成してください。」などで回復手段を示す

## 7. コピー

- 用語: 既存の「生成」「再生成」「WordPack」に統一

- エラー文: 何を確認できなかったか、次に何をすればよいかを明示
- 空状態: 変更なし
- disabled: プレビュー解決中の既存 `aria-busy` を維持
- トーン: ユーザーを責めず、「保存済みWordPackを確認できませんでした」と事実と回復手段を示す

## 8. 熟練者効率

- 主要反復タスク: 再生成後に結果を確認する
- 手数: 古い進行中カードを手で消して再検索する手間を削減
- 再入力・再選択: `wordPackId` と `jobId` から job status を確認するため再入力不要
- 近道: 完了補正後は既存のプレビュー導線をそのまま使える
- 初心者向け説明の影響: 常時説明やモーダルを追加していないため、熟練者の反復操作を妨げない
- 判定: Pass

## 9. 満足感・信頼感

- 待機中: 20分未満の進行中カードは既存の待機表示を維持
- 成功時: persisted job status が `succeeded` かつ result snapshot がある場合だけ成功へ補正する
- 失敗時: 確認不能な場合は成功扱いせず、一覧更新または再生成を案内する
- 危険操作: 破壊的操作なし。履歴消去は既存挙動のまま
- データ・権限・個人情報: 保存済みWordPack確認のみで、追加送信や公開操作はない
- トーン: 「必要なら」として過剰な不安を煽らない
- 判定: Pass

## 10. 反証レビュー

- 実装を落とす観点で見つけた問題: 既存 `wordPackId` の存在確認だけだと再生成前の内容を成功扱いするため、job status と result snapshot を確認できた場合だけ成功へ更新する実装にした
- P0候補: なし。エラー時に原因・影響・回復手段を示している

- 残リスク: `jobId` が保存されていない過去の進行中通知は成功補正できず、確認不能の失敗カードへ移る
